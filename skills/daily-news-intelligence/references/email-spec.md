# Email Spec — Gmail SMTP Delivery of Daily News Reports

Loaded at the end of the Workflow (Step 10) when the user passes `--email`. Describes the auto-generated subject, body template, attachment selection, and failure handling for `scripts/send-report-email.py`.

## When This Step Runs

Only when the `email` parameter is non-empty. If `email` is empty or missing, skip this step silently — local files are still delivered per Step 8 and Step 9.

## Environment Variables Required

Claude's Bash environment must have these exported (typically via `~/.zshrc` or `~/.bash_profile`):

| Variable | Required | Default | Purpose |
|---|---|---|---|
| `GOOGLE_EMAIL_USERNAME` | Yes | — | Gmail address (e.g. `you@gmail.com`) |
| `GOOGLE_EMAIL_APP_PASSWORD` | Yes | — | 16-char Gmail App Password |
| `GOOGLE_EMAIL_FROM_NAME` | No | = `USERNAME` | Display name in From header |
| `GOOGLE_EMAIL_HOST` | No | `smtp.gmail.com` | SMTP host |
| `GOOGLE_EMAIL_PORT` | No | `587` | SMTP port |
| `GOOGLE_EMAIL_START_TLS` | No | `true` | STARTTLS on port 587 |

See `.env.example` at the repo root.

**Never log, echo, or print the app password.** The skill prompt must instruct the orchestrator not to read these variables into its own output; they flow only through the Python subprocess environment.

## Subject Template

Default (override via `--email-subject`):

```
{country_display} {title_label} — {date_display}
```

Rendered per `lang` Localisation Table:

- `lang=zh` → `中国每日热点新闻 — 2026年4月14日`
- `lang=en` → `China Daily News Intelligence — April 14, 2026`
- `lang=ja` → `中国デイリーニュース — 2026年4月14日`

### Bilingual subject (1.18.0+)

When `--lang` is a two-language combo (`zh+en` / `en+zh` / `zh+ja` / `ja+zh` / `en+ja` / `ja+en`), the subject is the **primary-lang subject** with a bilingual tag appended:

```
{country_display_primary} {title_label_primary} — {date_display_primary} ({tag})
```

| `en+zh` | `China Daily News Intelligence — April 14, 2026 (Bilingual EN+ZH)` |

### Bilingual body (1.18.0+)

When `--lang` is a two-language combo, the body **stacks** the primary-lang single-lang body, then a divider, then the secondary-lang single-lang body. Both halves draw the same counts from the same Verifier Output (upstream ran once), but render `{country_display}` / `{date_display}` / `{attachment_names}` / category labels per their respective lang.

Layout (lang-agnostic skeleton):

```
{primary_lang_body — same shape as the single-lang `lang={primary}` template above,
 with {attachment_names} listing ALL files attached to this email — not just the
 primary-lang pair}

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

{secondary_lang_body — same shape as the single-lang `lang={secondary}` template
 above, with {attachment_names} listing ALL files attached to this email}
```

Notes on the bilingual body:
- The **same divider line** is used regardless of language combo: a run of `━` U+2501 (40 chars) on its own line, surrounded by blank lines.
- **Counts on both halves are identical** — they come from the same single Verifier run; only the labels and section headers translate.
- The `· 附件: / · Attachments:` section lists **every file in the email**, not only this half's pair — so the reader sees the full deliverable in either language.
- For 3-language combos (not supported in 1.18.0), the layout would extend with two dividers and three sections. Reserve that design for a later release.

## Attachment Selection

Controlled by `email_attach` parameter. Bilingual mode (`--lang zh+en` etc.) doubles the file count because each lang produces its own `.md` / `.docx` pair.

| Value | Single-lang behaviour | Bilingual behaviour (1.18.0+) |
|---|---|---|
| `both` (default) | Attach `{out_md}` + `{out_docx}` (2 files) | Attach all four: `{out_md_primary}`, `{out_docx_primary}`, `{out_md_secondary}`, `{out_docx_secondary}` (4 files) |
| `docx` | Attach only `{out_docx}` (1 file) | Attach `{out_docx_primary}` + `{out_docx_secondary}` (2 files) |
| `md` | Attach only `{out_md}` (1 file) | Attach `{out_md_primary}` + `{out_md_secondary}` (2 files) |
| `none` | Send no attachments (body-only email) | Send no attachments (body-only email) |

## Invocation

The skill orchestrator invokes the Python script via Bash. Write the body to a temp file first to avoid shell-quoting issues with multi-line content and Unicode quotes.

### Single-lang invocation

```bash
# Create temp body file
BODY_FILE=$(mktemp /tmp/daily-news-body.XXXXXX.txt)
cat > "$BODY_FILE" <<'EOF'
<body content here>
EOF

# Determine attachments based on email_attach parameter
case "$email_attach" in
  both) ATTACHMENTS=("$out_md" "$out_docx") ;;
  docx) ATTACHMENTS=("$out_docx") ;;
  md)   ATTACHMENTS=("$out_md") ;;
  none) ATTACHMENTS=() ;;
esac

# Call the sender. Do not pass --attach when ATTACHMENTS is empty.
if ((${#ATTACHMENTS[@]})); then
  ATTACH_ARGS=(--attach "${ATTACHMENTS[@]}")
else
  ATTACH_ARGS=()
fi

python3 "$PLUGIN_ROOT/scripts/send-report-email.py" \
  --to "$email" \
  --subject "$email_subject" \
  --body-file "$BODY_FILE" \
  "${ATTACH_ARGS[@]}"

SEND_EXIT=$?
rm -f "$BODY_FILE"
```

### Bilingual invocation (1.18.0+)

Same shape — only the attachment-array construction doubles, and the body file holds the stacked primary+secondary body:

```bash
BODY_FILE=$(mktemp /tmp/daily-news-body.XXXXXX.txt)
cat > "$BODY_FILE" <<'EOF'
<primary-lang body>

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

<secondary-lang body>
EOF

# Bilingual variant — both per-lang file pairs
case "$email_attach" in
  both) ATTACHMENTS=("$out_md_primary" "$out_docx_primary" "$out_md_secondary" "$out_docx_secondary") ;;
  docx) ATTACHMENTS=("$out_docx_primary" "$out_docx_secondary") ;;
  md)   ATTACHMENTS=("$out_md_primary" "$out_md_secondary") ;;
  none) ATTACHMENTS=() ;;
esac

if ((${#ATTACHMENTS[@]})); then
  ATTACH_ARGS=(--attach "${ATTACHMENTS[@]}")
else
  ATTACH_ARGS=()
fi

python3 "$PLUGIN_ROOT/scripts/send-report-email.py" \
  --to "$email" \
  --subject "$email_subject" \
  --body-file "$BODY_FILE" \
  "${ATTACH_ARGS[@]}"

SEND_EXIT=$?
rm -f "$BODY_FILE"
```

`send-report-email.py` accepts a plain-text `--body-file` with zero attachments. When `email_attach=none`, omit `--attach` entirely; the resulting email contains only the generated body.

If `--email-dry-run` is set, append `--dry-run` to the Python call. The script will print a summary and exit 0 without connecting to SMTP.

## Exit Code Handling

| Python script exit | Orchestrator action |
|---|---|
| 0 | Print "✅ email sent to {N} recipient(s)" |
| 1 | Print "⚠️ email skipped: missing env vars {list}. Local files are preserved." — do NOT delete reports |
| 2 | Print "⚠️ email skipped: invalid recipient format." — do NOT delete reports |
| 3 | Print "⚠️ email failed: SMTP error {message}. Local files are preserved." — do NOT delete reports |
| 4 | Print "⚠️ email failed: attachment not found. Local files are preserved." — do NOT delete reports |
| 5 | Print "⚠️ email failed: body file not found." — do NOT delete reports |
| 7 | Print "⚠️ email blocked: attachment has empty filename stem (e.g. `.docx`). Check `out_md` / `out_docx` construction per `language-spec.md` § Filename Pattern. Re-derive the filename and retry." |
| 8 | Print "⚠️ email blocked: attachment has no file extension. Check pandoc output or filename slug. Re-derive and retry." |
| 9 | Print "⚠️ email failed: Content-Disposition header is missing dual-filename encoding (script-internal regression). Inspect recent changes to `scripts/send-report-email.py § build_message` — both `filename=` (RFC 2047) and `filename*=` (RFC 2231) must be present. Local files are preserved." |

**Hard rule**: email failure never deletes, moves, or overwrites the local `.md` or `.docx` files. The report has already been delivered by Step 8 and Step 9.

## Security

- Never `echo`, `cat`, or interpolate `$GOOGLE_EMAIL_APP_PASSWORD` into any command that prints to the terminal.
- The Python subprocess inherits env vars implicitly — do NOT pass the password via `--flag` argument (it would show in process lists).
- Do not write the password to the temp body file, log files, or any stdout.
- Validate recipient addresses with the regex `^[^@\s]+@[^@\s]+\.[^@\s]+$` before sending (the Python script does this; the orchestrator does not need to duplicate the check).
- If you plan to share logs, scrub any accidental leak of `GOOGLE_EMAIL_USERNAME` too.

## Testing

Recommended order for first-time setup:

1. Verify env vars are set (without leaking passwords):
   ```bash
   [ -n "$GOOGLE_EMAIL_USERNAME" ] && echo "username: set" || echo "username: MISSING"
   [ -n "$GOOGLE_EMAIL_APP_PASSWORD" ] && echo "password: set" || echo "password: MISSING"
   ```
2. Dry-run the full pipeline:
   ```
   /daily-news-intelligence --country China --email you@gmail.com --email-dry-run
   ```
3. Send to yourself for inspection:
   ```
   /daily-news-intelligence --country China --email you@gmail.com
   ```
4. Check Gmail inbox — confirm subject, body render, attachments open correctly.
